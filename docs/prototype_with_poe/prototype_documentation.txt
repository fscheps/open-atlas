Project Overview
Mariner's Atlas is a single-file web application for creating interactive maritime route maps. Users can place ports, chart realistic sea routes using A* pathfinding that avoids land masses, annotate ports with draggable info bubbles, and export their work as PNG images or JSON files for backup/sharing.

Key Capabilities:

Interactive map with clickable port placement
Realistic maritime route calculation using A* pathfinding on a rasterized land/sea grid
Draggable annotation callout bubbles attached to ports
Port search and jump-to-location workflow
Straight-line distance ruler with km and nautical mile readouts
PNG export (map only, no UI controls)
JSON export/import for full state persistence
Responsive design with light/dark mode support

Core Components
1. Sea Routing Engine
Algorithm: A Pathfinding with Longitude Wraparound*

Grid Setup:

Resolution: 720 columns × 360 rows (0.5° per cell)
Coordinate mapping: latLngToCell(lat, lng) → grid cell [x, y]
Reverse mapping: cellToLatLng(x, y) → [lat, lng]
Longitude wraparound: x = (x + GRID_W) % GRID_W for antimeridian crossing
Rasterization Process:

Fetch TopoJSON land data from unpkg (110m simplified)
Convert to GeoJSON using topojson.feature()
Draw polygons to a 720×360 canvas using canvas.fillStyle
Read pixel data: white = land, black = sea
Store in Uint8Array isLand[720×360]
Manually carve canals with carveCanal() function:
Panama Canal (9.35°N, -79.92°W to 9.10°N, -79.55°W)
Suez Canal (30.55°N, 32.33°E to 29.95°N, 32.58°E)
Bosphorus, Dardanelles, Gibraltar, Bab-el-Mandeb, Hormuz, Malacca, Øresund
A Implementation:*

javascript
function astar(startX, startY, goalX, goalY, penaltyMap)

Data structures:
MinHeap (custom binary heap for priority queue)
Float32Array g[GRID_W * GRID_H] (cost-to-reach)
Int32Array came[GRID_W * GRID_H] (parent tracking)
Uint8Array closed[GRID_W * GRID_H] (visited set)
Movement: 8-directional (cardinal + diagonal)
Cost: 1 for cardinal, √2 for diagonal
Diagonal blocking: prevents squeezing between land corners
Heuristic: Euclidean distance with longitude wraparound
Penalty map: Optional Float32Array for alternative route search
Returns: Array of cell indices forming the path (or null if no route)
Port Snapping:

javascript
function nearestSea(x, y, maxRadius = 12)

Ports near coastlines may land on rasterized land pixels
BFS spiral search (max 12-cell radius) to find nearest sea cell
If unreachable, route drawing fails with error hint
Alternative Route Detection:

javascript
function findAlternative(start, goal, primaryCells)

Runs second A* with penalty corridor around primary path
Penalty: 20 * (1 - distance_from_primary / radius) within 4-cell radius
Comparison: If cellOverlap < 0.75 and altKm < primaryKm * 2.2, offer choice
User selects via modal with distance labels
Path Simplification:

Douglas-Peucker algorithm with tolerance 0.4° (reduces polyline vertices)
Smooths jaggy grid-based paths
Longitude Unwrapping:

javascript
function cellsToLatLngs(cells, hintLng)

Converts cell path to lat/lng coordinates
Continuously adjusts longitude to avoid 180° jumps (for Pacific routes)
Allows polylines to extend beyond [-180, 180] range
Leaflet's worldCopyJump handles rendering across antimeridian

4. Route Drawing
User Flow:

Port markers pulse with animation, hint displays "Click two ports to chart a sea route"

A* runs asynchronously (with setTimeout to yield to UI)
If alternative exists and differs significantly → route choice modal

5. Export System
PNG Export:

javascript
async function exportPNG()

Adds .exporting class to <body> (hides control panel, modals, Leaflet controls)
Waits 150ms for style changes to apply
Calls html2canvas(document.getElementById('map'), {useCORS: true, ...})
Tile layer has crossOrigin: 'anonymous' to allow canvas rendering
Converts canvas to PNG blob
Triggers download via temporary <a download> element
Filename: mariners-atlas-YYYY-MM-DDTHH-MM-SS.png
Removes .exporting class, restores UI
Known Limitations:

Requires CORS-enabled tile server (CartoDB Voyager supports this)
Leaflet permanent tooltips (port labels) render as DOM elements and are captured
Info bubbles (also DOM-based Leaflet markers) are captured
Scale: up to 2× device pixel ratio for high-DPI displays
JSON Export:

javascript
function exportJSON()

Serializes full state to JSON (see schema above)

JSON Import:

javascript
async function importJSONFile(file)

Parses JSON, validates format === 'mariners-atlas'
Calls clearAll() to remove existing ports/routes
Iterates data.ports[], calls addMarker() with original IDs
Recreates bubbles if bubbleVisible === true with saved bubbleLatLng
Iterates data.routes[], calls commitRoute() with stored polylines
Restores camera view with map.setView()
Shows hint with import summary

Deployment Constraints
CORS Requirements
Why a Server is Required:

Fetching unpkg.com/world-atlas@2/land-110m.json requires HTTP origin
CartoDB tile server sends CORS headers only for HTTP/HTTPS requests
file:// protocol has null origin, blocked by all external resources

Performance Characteristics
Load Time:

Initial HTML parse: <100ms
External script loading (Leaflet, TopoJSON, html2canvas): ~500ms (cached: ~50ms)
Sea grid build (fetch + rasterize + carve canals): 1-3 seconds
Total to interactive: ~2-4 seconds on broadband
A Pathfinding:*

Grid size: 720×360 = 259,200 cells
Average route (e.g., New York to London): ~200-500ms
Long route (e.g., New York to Tokyo): ~1-2 seconds
Worst-case (circumnavigation): ~3-4 seconds
Memory Usage:

Sea grid (Uint8Array): 259 KB
A* working memory (per run): ~6 MB (g, came, closed arrays)
Leaflet layers: ~50-100 KB per route
Typical session (10 ports, 5 routes): ~10 MB total
Optimization Opportunities:

Web Worker for A* computation (prevent UI freeze on long routes)
IndexedDB caching of TopoJSON data (skip re-fetch on reload)
Higher-resolution grid (1440×720) for narrow straits, at cost of 4× memory + slower pathfinding
Tile-based route rendering for very long routes (reduce polyline vertex count)
Known Limitations & Edge Cases
Straight-Line Fallback: None implemented. If sea routing fails, user sees error hint. Could add Euclidean fallback for debug/testing.

Antimeridian Routes: Handled via longitude unwrapping, but user may see polyline "wrapping" visually if bubble is dragged far from port across date line.

Polar Routes: Grid resolution degrades near poles (Mercator distortion). Routes above 80°N/S may be inaccurate.

Canal Precision: Manually carved canals are 1-2 cells wide (0.5-1° ~50-100km). Real-world canal widths are much narrower, but rasterization requires this.

Island Hopping: 110m simplification omits small islands. Routes may pass "through" visually, though pathfinding correctly avoids rasterized land.

Port Label Overlap: With many ports in small area, permanent labels overlap. No collision detection or label culling.

Bubble Pointer Direction: Always points from bottom-left corner. Doesn't rotate based on port position. Trade-off for simplicity vs. perfect UX.

Undo/Redo: Not implemented. Users must manually delete/recreate or restore from JSON export.

Distance Ruler Persistence: Measurements are intentionally excluded from JSON export/import so shared atlas files stay focused on ports and routes.

Multi-Port Routes: Not supported. Each route is point-to-point. Waypoint chains would require route chaining logic.
